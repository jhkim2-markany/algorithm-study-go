# 동적 프로그래밍

## 개념

동적 프로그래밍(Dynamic Programming, DP)은 복잡한 문제를 여러 개의 작은 하위 문제(subproblem)로 분해하여 풀고, 그 결과를 저장해 두었다가 동일한 하위 문제가 다시 등장할 때 재활용하는 알고리즘 설계 기법이다. 단순 재귀로 풀면 같은 하위 문제를 반복 계산하여 지수적 시간이 걸리는 문제를, DP를 적용하면 다항 시간 내에 해결할 수 있다.

DP가 적용 가능한 문제는 다음 두 가지 성질을 만족해야 한다:

- **최적 부분 구조(Optimal Substructure)**: 큰 문제의 최적해가 작은 하위 문제들의 최적해로부터 구성될 수 있다. 즉, 하위 문제를 최적으로 풀면 전체 문제도 최적으로 풀린다.
- **중복 부분 문제(Overlapping Subproblems)**: 재귀적으로 문제를 풀 때 동일한 하위 문제가 여러 번 반복하여 등장한다. 이 결과를 저장하면 중복 계산을 피할 수 있다.

DP의 핵심은 **상태(state)**를 정의하고, 상태 간의 관계를 나타내는 **점화식(recurrence relation, 상태 전이식)**을 세우는 것이다. 상태를 올바르게 정의하면 문제의 구조가 명확해지고, 점화식을 통해 체계적으로 해를 구할 수 있다.

## 동작 원리

### 탑다운 (Top-Down, 메모이제이션)

탑다운 방식은 재귀를 기반으로 하며, 이미 계산한 결과를 메모(memo) 테이블에 저장하여 중복 계산을 방지한다.

1. 큰 문제에서 출발하여 재귀적으로 하위 문제를 호출한다.
2. 하위 문제의 결과가 메모 테이블에 이미 존재하면 저장된 값을 즉시 반환한다.
3. 존재하지 않으면 재귀적으로 계산한 뒤 결과를 메모 테이블에 저장하고 반환한다.
4. 필요한 하위 문제만 계산하므로, 모든 상태를 채울 필요가 없는 경우 효율적이다.

### 바텀업 (Bottom-Up, 타뷸레이션)

바텀업 방식은 반복문을 기반으로 하며, 가장 작은 하위 문제부터 순서대로 테이블을 채워 나간다.

1. 기저 사례(base case)의 값을 DP 테이블에 초기화한다.
2. 작은 하위 문제부터 큰 문제 방향으로 점화식을 적용하여 테이블을 채운다.
3. 최종적으로 원래 문제에 해당하는 테이블 값이 답이 된다.
4. 재귀 호출 오버헤드가 없고, 스택 오버플로 위험이 없다.

### 상태 전이 (State Transition)

DP 문제를 푸는 핵심 과정은 다음과 같다:

1. **상태 정의**: 하위 문제를 표현하는 변수를 결정한다 (예: `dp[i]`, `dp[i][j]`).
2. **점화식 수립**: 현재 상태를 이전 상태들로부터 계산하는 관계식을 세운다.
3. **기저 사례 설정**: 더 이상 분해할 수 없는 가장 작은 하위 문제의 값을 설정한다.
4. **계산 순서 결정**: 탑다운 또는 바텀업 방식을 선택하여 모든 상태를 계산한다.
5. **최종 답 추출**: 원래 문제에 해당하는 상태 값을 반환한다.

## 복잡도

| 구분 | 복잡도 |
| --- | --- |
| 시간 복잡도 (최선) | O(상태 수 × 전이 비용) |
| 시간 복잡도 (평균) | O(상태 수 × 전이 비용) |
| 시간 복잡도 (최악) | O(상태 수 × 전이 비용) |
| 공간 복잡도 | O(상태 수) |

DP의 시간 복잡도는 "상태의 총 개수 × 각 상태를 계산하는 데 드는 비용"으로 결정된다. 예를 들어 피보나치 수는 상태 수 N, 전이 비용 O(1)이므로 O(N)이다. 0/1 배낭 문제는 상태 수 N×W, 전이 비용 O(1)이므로 O(NW)이다. 공간 복잡도는 DP 테이블 크기에 비례하며, 슬라이딩 윈도우 기법으로 최적화할 수 있는 경우도 있다.

## 적합한 문제 유형

- 피보나치 수열, 계단 오르기 등 수열/카운팅 문제
- 0/1 배낭 문제, 동전 교환 등 최적화 문제
- 최장 공통 부분 수열(LCS), 최장 증가 부분 수열(LIS)
- 편집 거리(Edit Distance), 문자열 매칭
- 행렬 체인 곱셈, 구간 DP
- 경로 탐색 (격자 위 최소 비용 경로)
- 확률/기댓값 DP
- 비트마스크 DP (상태 압축)


## 단계별 추적 (Trace)

### 예시 입력

0/1 배낭 문제: 배낭 용량 W=7, 물건 4개

```
물건  무게(w)  가치(v)
 1      1       1
 2      3       4
 3      4       5
 4      5       7
```

### 실행 과정

**상태 정의:** `dp[i][w]` = 물건 1~i까지 고려하고 배낭 용량이 w일 때의 최대 가치

**점화식:** `dp[i][w] = max(dp[i-1][w], dp[i-1][w-w_i] + v_i)` (w ≥ w_i인 경우)

```
DP 테이블 채우기 (바텀업):

        w=0  w=1  w=2  w=3  w=4  w=5  w=6  w=7
초기     0    0    0    0    0    0    0    0
물건1    0    1    1    1    1    1    1    1
물건2    0    1    1    4    5    5    5    5
물건3    0    1    1    4    5    6    6    9
물건4    0    1    1    4    5    7    8    9

물건2, w=3: max(dp[1][3], dp[1][0]+4) = max(1, 0+4) = 4
물건2, w=4: max(dp[1][4], dp[1][1]+4) = max(1, 1+4) = 5
물건3, w=7: max(dp[2][7], dp[2][3]+5) = max(5, 4+5) = 9
물건4, w=5: max(dp[3][5], dp[3][0]+7) = max(6, 0+7) = 7
물건4, w=7: max(dp[3][7], dp[3][2]+7) = max(9, 1+7) = 9
```

**역추적으로 선택된 물건 확인:**

```
dp[4][7] = 9, dp[3][7] = 9 → 물건4 선택 안 함
dp[3][7] = 9, dp[2][7] = 5 → 물건3 선택! (w=7-4=3으로 이동)
dp[2][3] = 4, dp[1][3] = 1 → 물건2 선택! (w=3-3=0으로 이동)
dp[1][0] = 0 → 물건1 선택 안 함

선택된 물건: 2, 3 (총 무게 7, 총 가치 9)
```

## 실전 팁

### 활용 노하우

- DP 문제를 만나면 "상태 정의 → 점화식 → 기저 사례 → 계산 순서" 순서로 접근한다
- 상태를 정의할 때 "이 상태만 알면 미래의 최적 결정을 내릴 수 있는가?"를 자문한다. 이것이 최적 부분 구조의 핵심이다
- 1차원 DP가 안 되면 2차원으로, 2차원이 안 되면 상태를 추가한다. 상태가 많아지면 비트마스크 DP를 고려한다
- 공간 최적화: 점화식이 이전 행만 참조하면 2개의 1차원 배열(또는 1개)로 줄일 수 있다. 배낭 문제에서 역순 순회로 1차원 배열만 사용하는 기법이 대표적이다

### 자주 하는 실수

- 점화식의 계산 순서를 잘못 설정하여 아직 계산되지 않은 값을 참조하는 실수. 바텀업에서는 작은 인덱스부터, 탑다운에서는 메모 테이블 확인을 먼저 한다
- 기저 사례를 빠뜨리거나 잘못 설정하여 잘못된 결과가 나오는 경우. `dp[0] = 0` 또는 `dp[0][0] = 0` 등을 명확히 초기화한다
- 1차원 배낭 DP에서 순방향으로 순회하면 같은 물건을 여러 번 선택하게 된다. 0/1 배낭은 역순, 완전 배낭은 순방향으로 순회한다
- 상태 전이에서 인덱스 범위를 벗어나는 접근: `w - w_i >= 0` 등의 조건을 반드시 확인한다

### 엣지 케이스 주의사항

- 배낭 용량이 0인 경우: 어떤 물건도 넣을 수 없으므로 가치 0
- 물건이 0개인 경우: 선택할 수 없으므로 가치 0
- 모든 물건의 무게가 배낭 용량보다 큰 경우: 아무것도 선택할 수 없다
- 음수 값이 포함된 경우: 초기값을 0이 아닌 음의 무한대로 설정해야 할 수 있다
- 답이 int 범위를 초과하는 경우: `int64`를 사용하거나 모듈러 연산을 적용한다

## 관련 알고리즘 비교

| 알고리즘 | 특징 | 적합한 상황 |
| --- | --- | --- |
| 동적 프로그래밍 | 중복 부분 문제 + 최적 부분 구조, 메모이제이션 | 배낭, LCS, LIS, 경로 탐색, 카운팅 |
| 분할 정복 | 독립적 부분 문제로 분할 후 결합 | 부분 문제가 겹치지 않는 경우 |
| 그리디 | 매 단계 최선의 선택, 되돌아가지 않음 | 탐욕 선택 성질이 보장되는 경우 |
| 브루트포스 | 모든 경우 탐색 | 탐색 공간이 작은 경우 |
| 백트래킹 | DFS + 가지치기 | 조합 탐색, 제약 조건 만족 문제 |

**DP vs 그리디 선택 기준:**
- 그리디가 최적해를 보장하는지 증명할 수 있으면 그리디를 사용한다 (더 빠르고 간단)
- 그리디가 최적해를 보장하지 않으면 DP를 사용한다
- 예: 동전 교환에서 동전 단위가 배수 관계이면 그리디, 아니면 DP
