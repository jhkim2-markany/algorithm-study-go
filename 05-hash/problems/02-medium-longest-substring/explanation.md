# 가장 긴 부분 문자열 - 해설

## 접근 방식

1. 슬라이딩 윈도우의 왼쪽 포인터(left)와 오른쪽 포인터(right)를 0에서 시작한다
2. 해시맵에 각 문자의 마지막 등장 위치를 저장한다
3. 오른쪽 포인터를 한 칸씩 이동하면서 현재 문자가 윈도우 내에 이미 존재하는지 확인한다
4. 중복이 발견되면 왼쪽 포인터를 중복 문자의 다음 위치로 이동한다
5. 매 단계에서 윈도우 크기의 최댓값을 갱신한다

## 핵심 아이디어

이 문제는 해시맵과 슬라이딩 윈도우를 결합하여 해결한다. 해시맵에 각 문자가 마지막으로 등장한 인덱스를 저장하면, 중복 문자를 만났을 때 왼쪽 포인터를 어디로 이동해야 하는지 O(1)에 결정할 수 있다.

핵심은 `left = max(left, lastSeen[ch] + 1)`이다. 중복 문자의 마지막 위치 다음으로 왼쪽 포인터를 이동하되, 현재 왼쪽 포인터보다 뒤로 가지 않도록 `max`를 취한다. 이렇게 하면 윈도우 내에 항상 중복 없는 문자만 유지된다.

## 복잡도 분석

| 구분 | 복잡도 |
| --- | --- |
| 시간 복잡도 | O(N) |
| 공간 복잡도 | O(min(N, 26)) |

오른쪽 포인터가 문자열을 한 번 순회하므로 시간 복잡도는 O(N)이다. 해시맵에는 최대 26개(알파벳 소문자)의 항목만 저장되므로 공간 복잡도는 O(1)로 볼 수 있다.

## 대안적 접근

- **브루트포스**: 모든 부분 문자열을 생성하고 중복 여부를 검사하는 O(N³) 방법이 있다. N이 100,000일 때 시간 초과가 발생한다.
- **해시셋 + 투 포인터**: 해시맵 대신 해시셋을 사용하여 윈도우 내 문자를 관리하고, 중복 시 왼쪽 포인터를 한 칸씩 이동하는 방법도 있다. 최악의 경우 O(2N)이지만 해시맵 방식보다 약간 느릴 수 있다.
