# 좌표 압축

## 개념

좌표 압축(Coordinate Compression)은 큰 범위의 좌표 값을 상대적인 순서를 유지하면서 작은 범위의 연속된 인덱스(0, 1, 2, ...)로 변환하는 기법이다. 좌표 값 자체가 아닌 값들 사이의 상대적 순서만 중요한 경우에 사용한다.

예를 들어, 좌표 값이 {100, 5000, 30, 5000, 100}인 경우 정렬 후 중복을 제거하면 {30, 100, 5000}이 되고, 각각 {0, 1, 2}로 매핑된다. 원래 배열은 {1, 2, 0, 2, 1}로 압축된다.

좌표 압축이 필요한 이유:

- 좌표 값의 범위가 매우 클 때(예: -10⁹ ~ 10⁹) 배열 인덱스로 직접 사용할 수 없다
- 세그먼트 트리, BIT(펜윅 트리) 등 인덱스 기반 자료구조를 사용하려면 좌표를 작은 범위로 압축해야 한다
- 값의 절대적 크기가 아닌 상대적 순서만 필요한 문제에서 메모리와 시간을 절약할 수 있다

## 동작 원리

좌표 압축은 정렬 + 중복 제거 + 이진 탐색의 세 단계로 이루어진다.

### 1. 좌표 수집 및 정렬

1. 압축할 모든 좌표 값을 하나의 배열에 수집한다
2. 배열을 오름차순으로 정렬한다

### 2. 중복 제거

1. 정렬된 배열에서 연속된 중복 값을 제거한다
2. 결과 배열의 각 원소는 고유한 좌표 값이 된다
3. 이 배열의 인덱스가 곧 압축된 좌표이다

### 3. 이진 탐색으로 매핑

1. 원래 좌표 값에 대해 중복 제거된 배열에서 이진 탐색을 수행한다
2. 찾은 인덱스가 해당 좌표의 압축된 값이 된다
3. Go에서는 `sort.SearchInts()` 함수를 활용할 수 있다

### 예시

원본 배열: [100, 5000, 30, 5000, 100]

1. 정렬: [30, 100, 5000]이 아닌, 원본을 복사하여 정렬 → [30, 100, 100, 5000, 5000]
2. 중복 제거: [30, 100, 5000]
3. 매핑:
   - 100 → 이진 탐색 → 인덱스 1
   - 5000 → 이진 탐색 → 인덱스 2
   - 30 → 이진 탐색 → 인덱스 0
4. 압축 결과: [1, 2, 0, 2, 1]

## 복잡도

| 구분 | 복잡도 |
| --- | --- |
| 시간 복잡도 (정렬) | O(N log N) |
| 시간 복잡도 (중복 제거) | O(N) |
| 시간 복잡도 (매핑, 이진 탐색 N회) | O(N log N) |
| 전체 시간 복잡도 | O(N log N) |
| 공간 복잡도 | O(N) |

정렬이 O(N log N)으로 가장 지배적이며, 이진 탐색을 N번 수행하는 것도 O(N log N)이다. 전체적으로 O(N log N)의 시간 복잡도를 가진다.

## 적합한 문제 유형

- 좌표 값의 범위가 매우 크지만 실제 사용되는 좌표 수는 적은 문제
- 세그먼트 트리나 BIT를 사용해야 하는데 좌표 범위가 큰 문제
- 값의 상대적 순서(랭킹)를 구하는 문제
- 2D 평면에서 좌표를 압축하여 격자로 변환하는 문제
- 구간 쿼리 문제에서 좌표 범위를 줄여야 하는 문제
- 스위핑(Sweeping) 알고리즘과 함께 사용하는 문제


## 단계별 추적 (Trace)

### 예시 1: 1차원 좌표 압축

```
입력 배열: [300, 50, 1000000, 50, 300, 7]

1단계: 좌표 수집 및 복사
  sorted = [300, 50, 1000000, 50, 300, 7]

2단계: 정렬
  sorted = [7, 50, 50, 300, 300, 1000000]

3단계: 중복 제거
  unique = [7, 50, 300, 1000000]
  인덱스:   0   1    2       3

4단계: 이진 탐색으로 매핑
  300     → 이진 탐색 → 인덱스 2
  50      → 이진 탐색 → 인덱스 1
  1000000 → 이진 탐색 → 인덱스 3
  50      → 이진 탐색 → 인덱스 1
  300     → 이진 탐색 → 인덱스 2
  7       → 이진 탐색 → 인덱스 0

결과: [2, 1, 3, 1, 2, 0]

원래 범위: 7 ~ 1,000,000 (약 100만)
압축 범위: 0 ~ 3 (4개 값)
```

### 예시 2: 2차원 좌표 압축 (세그먼트 트리 활용)

```
입력 좌표: (100, 500), (200, 300), (100, 300), (500, 500)

1단계: x좌표와 y좌표를 각각 수집
  x좌표: [100, 200, 100, 500]
  y좌표: [500, 300, 300, 500]

2단계: 각각 정렬 + 중복 제거
  x_unique = [100, 200, 500]  → 인덱스 0, 1, 2
  y_unique = [300, 500]       → 인덱스 0, 1

3단계: 매핑
  (100, 500) → (0, 1)
  (200, 300) → (1, 0)
  (100, 300) → (0, 0)
  (500, 500) → (2, 1)

압축 전 격자: 100 ~ 500 (최대 401×201)
압축 후 격자: 3 × 2 (6칸)

시각화:
  압축 전:              압축 후:
  y                     y
  500 | *     *         1 | *  *
  300 | *  *            0 | *  *
      +--------→ x        +-----→ x
       100 200 500          0  1  2
```

## 실전 팁

### 활용 노하우

- 좌표 압축은 세그먼트 트리, BIT와 함께 사용하는 경우가 대부분이다. 좌표 범위가 크면 먼저 압축을 떠올리자
- Go에서는 `sort.Ints()` + 직접 중복 제거 + `sort.SearchInts()`로 간결하게 구현할 수 있다
- 2차원 문제에서는 x좌표와 y좌표를 독립적으로 압축한다
- 스위핑 알고리즘과 함께 사용할 때, 이벤트 좌표만 수집하여 압축하면 된다
- 압축 후 인덱스가 0부터 시작하므로, 1-indexed 자료구조를 사용할 때는 +1을 해주자

### 자주 하는 실수

- 원본 배열을 직접 정렬하여 원래 순서를 잃어버리는 실수 → 반드시 복사본을 정렬한다
- 중복 제거를 하지 않아 같은 값이 다른 인덱스로 매핑되는 실수
- 이진 탐색 대신 선형 탐색을 사용하여 시간 초과가 나는 실수
- 좌표 압축 후 원래 값이 필요한데 역매핑을 준비하지 않는 실수
- 쿼리의 좌표도 함께 압축해야 하는데 배열 값만 압축하는 실수

### 엣지 케이스

- 모든 좌표가 같은 경우: 압축 후 모두 인덱스 0
- 좌표가 1개인 경우: 압축 후 인덱스 0
- 음수 좌표가 포함된 경우: 정렬과 이진 탐색은 음수도 정상 처리
- 좌표 값이 int64 범위인 경우: 정렬과 비교 연산에서 오버플로 주의
- 좌표 수가 매우 많은 경우 (10^6): 정렬이 O(N log N)이므로 충분히 빠름

## 관련 알고리즘 비교

| 알고리즘 | 시간 복잡도 | 적합한 상황 | 특징 |
| --- | --- | --- | --- |
| 좌표 압축 | O(N log N) | 큰 범위 → 작은 인덱스 변환 | 전처리 기법, 단독 사용 드묾 |
| 해시맵 매핑 | O(N) 평균 | 값 → 인덱스 매핑 | 순서 보장 안 됨 |
| 이산화 (Discretization) | O(N log N) | 좌표 압축과 동일 | 좌표 압축의 다른 이름 |
| 세그먼트 트리 + 좌표 압축 | O(N log N) | 구간 쿼리 + 큰 좌표 | 가장 흔한 조합 |
| BIT + 좌표 압축 | O(N log N) | 누적합 쿼리 + 큰 좌표 | 구현이 더 간단 |

**선택 기준:**
- 인덱스 기반 자료구조 사용 + 좌표 범위가 큼 → 좌표 압축 필수
- 순서가 중요하지 않고 존재 여부만 필요 → 해시맵으로 충분
- 구간 쿼리가 필요 → 좌표 압축 + 세그먼트 트리 (25번 참조)
- 역전 쌍 세기 → 좌표 압축 + BIT
- 스위핑 문제 → 좌표 압축 + 이벤트 정렬
