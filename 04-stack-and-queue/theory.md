# 스택과 큐

## 개념

스택(Stack)은 후입선출(LIFO, Last In First Out) 원칙을 따르는 선형 자료구조이다. 가장 나중에 삽입된 데이터가 가장 먼저 제거된다. 접시를 쌓아 올리는 것과 같은 구조로, 맨 위(top)에서만 삽입과 삭제가 이루어진다.

큐(Queue)는 선입선출(FIFO, First In First Out) 원칙을 따르는 선형 자료구조이다. 가장 먼저 삽입된 데이터가 가장 먼저 제거된다. 줄을 서서 기다리는 것과 같은 구조로, 뒤(rear)에서 삽입하고 앞(front)에서 삭제한다.

### 스택의 주요 연산

- **Push**: 스택의 맨 위에 원소를 추가한다
- **Pop**: 스택의 맨 위 원소를 제거하고 반환한다
- **Peek/Top**: 스택의 맨 위 원소를 제거하지 않고 확인한다
- **IsEmpty**: 스택이 비어 있는지 확인한다

### 큐의 주요 연산

- **Enqueue**: 큐의 뒤에 원소를 추가한다
- **Dequeue**: 큐의 앞 원소를 제거하고 반환한다
- **Front**: 큐의 앞 원소를 제거하지 않고 확인한다
- **IsEmpty**: 큐가 비어 있는지 확인한다

## 동작 원리

### 스택의 동작

1. 빈 스택을 생성한다
2. Push 연산 시 top 포인터를 증가시키고 해당 위치에 데이터를 저장한다
3. Pop 연산 시 top 위치의 데이터를 반환하고 top 포인터를 감소시킨다
4. top 포인터가 -1이면 스택이 비어 있는 상태이다

### 큐의 동작

1. 빈 큐를 생성한다
2. Enqueue 연산 시 rear 포인터를 증가시키고 해당 위치에 데이터를 저장한다
3. Dequeue 연산 시 front 위치의 데이터를 반환하고 front 포인터를 증가시킨다
4. front와 rear가 같은 위치이면 큐가 비어 있는 상태이다

### Golang에서의 구현

Go에서는 슬라이스를 활용하여 스택과 큐를 간단하게 구현할 수 있다:
- **스택**: 슬라이스의 `append`로 Push, 마지막 원소 접근 후 슬라이싱으로 Pop
- **큐**: 슬라이스의 `append`로 Enqueue, 첫 번째 원소 접근 후 슬라이싱으로 Dequeue

## 복잡도

| 구분 | 복잡도 |
| --- | --- |
| 시간 복잡도 (Push/Enqueue) | O(1) |
| 시간 복잡도 (Pop/Dequeue) | O(1) (amortized) |
| 시간 복잡도 (Peek/Front) | O(1) |
| 공간 복잡도 | O(N) |

스택과 큐의 모든 기본 연산은 O(1)에 수행된다. Go 슬라이스 기반 큐의 Dequeue는 슬라이싱으로 O(1)이지만, 메모리 해제 관점에서는 별도 처리가 필요할 수 있다. N개의 원소를 저장하므로 공간 복잡도는 O(N)이다.

## 적합한 문제 유형

- 괄호 짝 맞추기, 수식 유효성 검사 (스택)
- 후위 표기법 계산, 중위→후위 변환 (스택)
- 히스토그램에서 가장 큰 직사각형 (스택)
- 주식 가격 스팬, 다음 큰 원소 찾기 (스택 - 모노톤 스택)
- BFS 탐색, 레벨 순서 순회 (큐)
- 프로세스 스케줄링, 작업 대기열 시뮬레이션 (큐)
- 슬라이딩 윈도우 최댓값/최솟값 (덱 - 양방향 큐)


## 단계별 추적 (Trace)

### 예시: 괄호 유효성 검사 (스택)

입력: `"({[]})"`

```
문자별 처리 과정:

문자 '(' → 여는 괄호 → Push → 스택: ['(']
문자 '{' → 여는 괄호 → Push → 스택: ['(', '{']
문자 '[' → 여는 괄호 → Push → 스택: ['(', '{', '[']
문자 ']' → 닫는 괄호 → Top='[' 매칭 ✓ → Pop → 스택: ['(', '{']
문자 '}' → 닫는 괄호 → Top='{' 매칭 ✓ → Pop → 스택: ['(']
문자 ')' → 닫는 괄호 → Top='(' 매칭 ✓ → Pop → 스택: []

스택 시각화:
  단계 1:  [(]
  단계 2:  [(, {]
  단계 3:  [(, {, []       ← 스택 최대 깊이
  단계 4:  [(, {]          ← ']'와 '[' 매칭
  단계 5:  [(]             ← '}'와 '{' 매칭
  단계 6:  []              ← ')'와 '(' 매칭

결과: 스택이 비어 있으므로 유효한 괄호 → true
```

### 예시: BFS 레벨 순서 탐색 (큐)

입력: 이진 트리

```
        1
       / \
      2   3
     / \   \
    4   5   6

큐를 이용한 레벨 순서 탐색:

초기: 큐 = [1]

레벨 0: Dequeue → 1, 자식 2,3 Enqueue → 큐 = [2, 3]
  출력: 1

레벨 1: Dequeue → 2, 자식 4,5 Enqueue → 큐 = [3, 4, 5]
         Dequeue → 3, 자식 6 Enqueue  → 큐 = [4, 5, 6]
  출력: 2 3

레벨 2: Dequeue → 4 (자식 없음) → 큐 = [5, 6]
         Dequeue → 5 (자식 없음) → 큐 = [6]
         Dequeue → 6 (자식 없음) → 큐 = []
  출력: 4 5 6

최종 출력: 1 / 2 3 / 4 5 6
```

## 실전 팁

### 활용 노하우

- **모노톤 스택 패턴**: "다음 큰 원소(Next Greater Element)" 유형은 스택에 인덱스를 저장하고, 현재 원소가 스택 top보다 크면 Pop하는 패턴이다. O(N)에 해결 가능하다
- **Go 슬라이스로 스택 구현**: `stack = append(stack, val)` (Push), `val = stack[len(stack)-1]; stack = stack[:len(stack)-1]` (Pop) 패턴을 외워두자
- **큐 대신 슬라이스 인덱스**: Go에서 큐를 슬라이스로 구현할 때, Dequeue마다 `queue = queue[1:]`을 하면 메모리 누수가 발생할 수 있다. 인덱스 변수 `front`를 사용하는 것이 더 효율적이다
- **괄호 문제 패턴**: 여는 괄호는 Push, 닫는 괄호는 Pop 후 매칭 확인. 최종적으로 스택이 비어 있으면 유효하다

### 자주 하는 실수

- **빈 스택에서 Pop**: Pop이나 Top 연산 전에 반드시 `len(stack) > 0`을 확인해야 한다. 빈 스택에서 접근하면 런타임 에러가 발생한다
- **큐의 메모리 관리**: Go 슬라이스 기반 큐에서 `queue[1:]`로 Dequeue하면 앞쪽 메모리가 해제되지 않는다. 대량 데이터 처리 시 인덱스 기반 구현을 사용하자
- **스택/큐 혼동**: 문제에서 LIFO가 필요한지 FIFO가 필요한지 정확히 파악하자. BFS는 큐, DFS는 스택이다
- **모노톤 스택에서 등호 처리**: `>=`와 `>`의 차이로 결과가 달라질 수 있다. 문제 조건을 정확히 확인하자

### 엣지 케이스 주의사항

- **빈 문자열**: 괄호 검사에서 빈 문자열은 유효한 것으로 처리해야 한다
- **닫는 괄호로 시작**: `")("` 같은 입력에서 스택이 비어 있을 때 Pop을 시도하면 에러가 발생한다
- **단일 원소**: 스택이나 큐에 원소가 1개일 때의 동작을 확인하자
- **모든 원소가 동일**: 모노톤 스택에서 동일한 값이 연속될 때의 처리를 확인하자

## 관련 알고리즘 비교

| 자료구조 | 접근 방식 | 삽입/삭제 | 적합한 상황 |
| --- | --- | --- | --- |
| 스택 (LIFO) | 한쪽 끝에서만 삽입/삭제 | O(1) | 괄호 매칭, 후위 표기법, DFS, 모노톤 스택 |
| 큐 (FIFO) | 뒤에서 삽입, 앞에서 삭제 | O(1) | BFS, 작업 스케줄링, 레벨 순서 탐색 |
| 덱 (Deque) | 양쪽 끝에서 삽입/삭제 | O(1) | 슬라이딩 윈도우 최대/최소, 0-1 BFS |
| 우선순위 큐 (Heap) | 우선순위가 높은 원소 먼저 | O(log N) | 다익스트라, K번째 원소, 중앙값 유지 |

**선택 기준**:
- 가장 최근에 추가된 것을 먼저 처리 → 스택
- 가장 먼저 추가된 것을 먼저 처리 → 큐
- 양쪽 끝에서 모두 접근 필요 → 덱
- 값의 크기 순서대로 처리 → 우선순위 큐
