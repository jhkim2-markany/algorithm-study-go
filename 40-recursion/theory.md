# 재귀

## 개념

재귀(Recursion)란 함수가 자기 자신을 호출하여 문제를 더 작은 부분 문제로 나누어 해결하는 프로그래밍 기법이다. 재귀 함수는 반드시 기저 조건(Base Case)을 가져야 하며, 각 재귀 호출은 문제의 크기를 줄여 기저 조건에 도달하도록 해야 한다.

재귀의 핵심 구성 요소는 다음과 같다:

- 기저 조건(Base Case): 재귀 호출을 멈추는 조건. 이것이 없으면 무한 재귀에 빠진다
- 재귀 단계(Recursive Step): 문제를 더 작은 부분 문제로 분할하여 자기 자신을 호출하는 부분
- 호출 스택(Call Stack): 각 재귀 호출의 상태(지역 변수, 반환 주소)를 저장하는 메모리 영역

재귀는 수학적 귀납법과 밀접한 관계가 있다. 기저 조건이 참이고, n번째 단계가 참이면 n+1번째도 참임을 보이는 것과 같은 원리로 동작한다.

## 선수 학습

재귀를 학습하기 전에 다음 내용을 먼저 학습하는 것을 권장한다:

- [18-divide-and-conquer](../18-divide-and-conquer/) - 분할 정복은 재귀의 대표적 응용으로, 문제를 분할하고 재귀적으로 해결한 뒤 결합하는 패턴을 학습할 수 있다

## 동작 원리

### 1. 재귀 함수의 기본 구조

재귀 함수는 다음과 같은 구조를 따른다:

1. 기저 조건을 확인한다 (문제가 충분히 작은지 판단)
2. 기저 조건이 만족되면 즉시 결과를 반환한다
3. 그렇지 않으면 문제를 더 작은 부분 문제로 나눈다
4. 부분 문제에 대해 자기 자신을 재귀 호출한다
5. 부분 문제의 결과를 조합하여 최종 결과를 반환한다

### 2. 호출 스택의 동작

재귀 호출 시 시스템 호출 스택에서 일어나는 과정:

1. 함수가 호출되면 새로운 스택 프레임이 생성된다
2. 스택 프레임에는 매개변수, 지역 변수, 반환 주소가 저장된다
3. 재귀 호출이 발생하면 새로운 스택 프레임이 위에 쌓인다
4. 기저 조건에 도달하면 결과를 반환하고 스택 프레임이 제거된다
5. 이전 호출로 돌아가 나머지 연산을 수행한다
6. 모든 스택 프레임이 제거되면 최종 결과가 반환된다

### 3. 꼬리 재귀 (Tail Recursion)

꼬리 재귀는 재귀 호출이 함수의 마지막 연산인 형태이다:

1. 일반 재귀: `return n * factorial(n-1)` — 재귀 호출 후 곱셈 연산이 남아 있다
2. 꼬리 재귀: `return factorial(n-1, n*acc)` — 재귀 호출이 마지막 연산이다
3. 꼬리 재귀는 컴파일러가 반복문으로 최적화할 수 있어 스택 오버플로를 방지한다
4. Go 언어는 꼬리 재귀 최적화를 보장하지 않지만, 개념적으로 이해하는 것이 중요하다

### 4. 메모이제이션과의 관계

재귀에서 동일한 부분 문제가 반복 계산되는 경우 메모이제이션을 적용한다:

1. 재귀 함수의 결과를 캐시(맵 또는 배열)에 저장한다
2. 동일한 입력으로 호출되면 캐시에서 결과를 즉시 반환한다
3. 이를 통해 지수 시간 복잡도를 다항 시간으로 줄일 수 있다
4. 메모이제이션이 적용된 재귀는 Top-down 동적 프로그래밍과 동일하다

## 복잡도

| 구분 | 복잡도 |
| --- | --- |
| 팩토리얼 (선형 재귀) | O(N) |
| 피보나치 (이진 재귀, 메모이제이션 없음) | O(2^N) |
| 피보나치 (메모이제이션 적용) | O(N) |
| 하노이의 탑 | O(2^N) |
| 부분집합 생성 | O(2^N) |
| 공간 복잡도 (호출 스택) | O(재귀 깊이) |

재귀의 시간 복잡도는 재귀 트리의 노드 수에 비례하며, 공간 복잡도는 재귀 깊이(호출 스택의 최대 깊이)에 비례한다.

## 적합한 문제 유형

- 수학적 점화식을 직접 구현하는 문제 (팩토리얼, 피보나치 등)
- 트리 구조를 순회하거나 탐색하는 문제
- 모든 경우의 수를 열거하는 문제 (순열, 조합, 부분집합)
- 분할 정복 패턴의 문제 (병합 정렬, 퀵 정렬 등)
- 하노이의 탑과 같은 고전적 재귀 문제
- 프랙탈 패턴 생성 문제
- 문자열이나 배열을 재귀적으로 처리하는 문제

## 단계별 추적 (Trace)

### 예시: 팩토리얼 factorial(4) 계산

```
호출: factorial(4)
  │  4 != 0 → 재귀 호출
  ├─ 호출: factorial(3)
  │    │  3 != 0 → 재귀 호출
  │    ├─ 호출: factorial(2)
  │    │    │  2 != 0 → 재귀 호출
  │    │    ├─ 호출: factorial(1)
  │    │    │    │  1 != 0 → 재귀 호출
  │    │    │    ├─ 호출: factorial(0)
  │    │    │    │    0 == 0 → 기저 조건! 반환 1
  │    │    │    │
  │    │    │    └─ 반환: 1 * 1 = 1
  │    │    └─ 반환: 2 * 1 = 2
  │    └─ 반환: 3 * 2 = 6
  └─ 반환: 4 * 6 = 24

최종 결과: 24
```

### 호출 스택 시각화

```
[단계 1] 호출 스택 (위가 top)
┌──────────────┐
│ factorial(4)  │  ← 현재 실행 중
└──────────────┘

[단계 2]
┌──────────────┐
│ factorial(3)  │  ← 현재 실행 중
├──────────────┤
│ factorial(4)  │  대기 중 (4 * ?)
└──────────────┘

[단계 3]
┌──────────────┐
│ factorial(2)  │  ← 현재 실행 중
├──────────────┤
│ factorial(3)  │  대기 중 (3 * ?)
├──────────────┤
│ factorial(4)  │  대기 중 (4 * ?)
└──────────────┘

[단계 4]
┌──────────────┐
│ factorial(1)  │  ← 현재 실행 중
├──────────────┤
│ factorial(2)  │  대기 중 (2 * ?)
├──────────────┤
│ factorial(3)  │  대기 중 (3 * ?)
├──────────────┤
│ factorial(4)  │  대기 중 (4 * ?)
└──────────────┘

[단계 5] 기저 조건 도달
┌──────────────┐
│ factorial(0)  │  → 반환 1
├──────────────┤
│ factorial(1)  │  대기 중 (1 * ?)
├──────────────┤
│ ...           │
└──────────────┘

[단계 6~9] 스택 되감기
factorial(0) → 1
factorial(1) → 1 * 1 = 1
factorial(2) → 2 * 1 = 2
factorial(3) → 3 * 2 = 6
factorial(4) → 4 * 6 = 24
```

### 예시: 피보나치 fib(5) 재귀 트리

```
                    fib(5)
                   /      \
              fib(4)       fib(3)
             /     \       /     \
         fib(3)  fib(2)  fib(2)  fib(1)
        /    \    /   \   /   \     |
    fib(2) fib(1) fib(1) fib(0) fib(1) fib(0)  1
    /   \    |     |      |      |      |
fib(1) fib(0) 1    1      0      1      0
  |      |
  1      0

총 호출 횟수: 15회 (중복 계산 다수)
fib(2)가 3번, fib(1)이 5번, fib(0)이 3번 호출됨
→ 메모이제이션 적용 시 각 값은 1번만 계산
```

## 실전 팁

### 활용 노하우

- 재귀 함수를 작성할 때 기저 조건을 먼저 작성하는 습관을 들이자. 기저 조건이 명확해야 무한 재귀를 방지할 수 있다
- Go에서 재귀 깊이가 깊어질 수 있는 경우, 반복문으로 변환하는 것을 고려하자. Go는 꼬리 재귀 최적화를 지원하지 않는다
- 재귀 함수의 매개변수에 현재 상태를 모두 담아 전달하면 디버깅이 쉬워진다
- 부분 문제가 겹치는 경우 반드시 메모이제이션을 적용하자. 피보나치처럼 O(2^N)이 O(N)으로 줄어든다
- 재귀 → 반복 변환 시 명시적 스택(슬라이스)을 사용하면 호출 스택 제한을 우회할 수 있다

### 자주 하는 실수

- 기저 조건 누락 또는 잘못된 기저 조건 설정으로 무한 재귀에 빠지는 경우
- 재귀 호출 시 문제 크기가 줄어들지 않아 종료되지 않는 경우
- Go의 기본 스택 크기(1MB → 자동 확장)를 과신하여 매우 깊은 재귀를 사용하는 경우. N이 10^5 이상이면 반복문 변환을 고려해야 한다
- 메모이제이션 없이 중복 부분 문제가 있는 재귀를 사용하여 시간 초과가 발생하는 경우
- 전역 변수를 재귀 함수 내에서 수정할 때 되돌리기(backtrack)를 잊는 경우

### 엣지 케이스

- 입력이 0이나 1인 경우 기저 조건에 해당하는지 확인
- 음수 입력이 들어올 수 있는 경우 기저 조건에서 처리
- 빈 배열이나 빈 문자열에 대한 재귀 호출
- 재귀 깊이가 N에 비례하는 경우 (편향 트리, 정렬된 입력 등) 스택 오버플로 가능성 확인
- 부분 문제의 결과가 0인 경우와 아직 계산하지 않은 경우를 구분해야 하는 메모이제이션 상황

## 관련 알고리즘 비교

| 알고리즘 | 특징 | 적합한 상황 |
| --- | --- | --- |
| 재귀 (일반) | 함수가 자기 자신을 호출, 호출 스택 사용 | 트리 순회, 수학적 점화식, 간결한 구현이 필요할 때 |
| 반복문 | 루프로 상태를 갱신, 스택 오버플로 없음 | 단순 반복 패턴, 깊이가 깊은 경우, 성능이 중요할 때 |
| 분할 정복 | 문제를 독립적 부분 문제로 분할 후 결합 | 병합 정렬, 퀵 정렬, 거듭제곱 등 부분 문제가 독립적일 때 |
| 동적 프로그래밍 (Top-down) | 재귀 + 메모이제이션 | 부분 문제가 겹치고, 최적 부분 구조가 있을 때 |
| 동적 프로그래밍 (Bottom-up) | 반복문으로 작은 문제부터 해결 | 부분 문제 순서가 명확하고, 스택 오버플로를 피해야 할 때 |
| 백트래킹 | 재귀 + 가지치기 | 모든 경우의 수 탐색에서 불필요한 탐색을 줄여야 할 때 |
